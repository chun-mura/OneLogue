# OneLogue — プロダクト概要

## 一行で

**「今やるべき1つ」に集中させる、排他タイマー内蔵のタスク × 時間計測アプリ。**

## コアバリュー

並行して走る通常のTODO/タイムトラッカーと違い、OneLogueは**同時に1つのタスクしかタイマーが走らない**仕組み (排他タイマー) を前提に設計されている。マルチタスクによる注意分散を構造的に防ぎ、「何にどれだけ時間を使ったか」を事後ではなく実行中に意識させる。

## ターゲット

- ソロワーカー / 個人事業主 / 研究者 / 学習者
- 作業分散で生産性が下がりやすい知識労働者
- Toggl Trackのような時間計測と、TodoistのようなTODO管理を**1つの画面で**回したいユーザー

## ユースケース

1. 朝、今日のタスクを並べる (期限・カテゴリ付与)
2. 最優先のタスクをStart → 自動でタイマー始動
3. 別タスクに切り替えるとき、新タスクをStart → 前タスクは**自動停止** (明示的Stop不要)
4. 日次 / 週次 / 月次でカテゴリ別の投下時間を集計ダッシュボードで確認

## 機能スコープ (MVP)

### タスク管理
- CRUD、期限 (日付 + 時刻)、カテゴリ、ステータス (pending / completed)
- 期限切れ / 今日 / 今後 / 完了済み の自動グルーピング
- ソート (期限昇降、作成日、タイトル)、カテゴリフィルタ、期限フィルタ (今日 / 7日以内 / 全て)

### 排他タイマー
- 1タスクのみ稼働保証 (サーバ側で一貫性担保)
- 経過時間の秒単位リアルタイム表示
- タスク切替時は既存計測を自動クローズ

### 時間ログ
- 過去の計測履歴を日付 / カレンダーで遡及閲覧
- 手動登録・編集対応

### 集計ダッシュボード
- 日次 / 週次 / 月次 / カスタム期間
- カテゴリ別の時間配分

### カテゴリ管理
- 自由定義のラベル体系

## アーキテクチャ

```
┌─────────────────────┐        ┌──────────────────────┐
│  Next.js 15 (React19)│  REST  │  FastAPI              │
│  App Router / CSR     │ ◄────►│  SQLAlchemy           │
│  Tailwind CSS         │        │  Pydantic             │
│  Asia/Tokyo固定TZ処理 │        │  TimerService (排他)  │
└─────────────────────┘        └──────────┬───────────┘
                                           │
                                      ┌────▼────┐
                                      │ SQLite  │
                                      │ (単一DB)│
                                      └─────────┘
```

- **Frontend**: Next.js 15 App Router + React 19 + Tailwind。クライアント主体レンダリング (`isMounted` ガード) + 独自の `AppTimeProvider` による全画面共通の「今」配信。
- **Backend**: FastAPI。ルータは `tasks` / `time_entries` / `categories` / `stats` の4系統。排他制御は `TimerService` に集約。
- **データモデル**: `Category` / `Task` / `TimeEntry` の3エンティティ。
- **デプロイ**: `docker compose up` で一発起動。データは `./data/db.sqlite3` に永続化。

## 設計思想

### 1. 排他性は「機能」ではなく「制約」
複数同時タイマーを許さないのはUI制限ではなくサーバ側の不変条件。クライアントが誤操作してもデータは一貫する。

### 2. タイムゾーン固定 (Asia/Tokyo)
グローバル対応より国内ユーザーの日付境界の正確さを優先。日付キー / カレンダー生成すべて +9 固定オフセットで実装、ロケール依存の曖昧性を排除。

### 3. ネイティブUIを避けた独自プルダウン
OS差異吸収と統一感のため、`custom-dropdown` を全セレクタの標準に。

## 現時点の制約

- **単一ユーザー前提** (認証なし、SQLite単一DB)
- 進行中タイマーはフロント側メモリ保持 → リロード時の再同期は未実装
- マルチデバイス同期・共有機能なし

## 成長余地 (ロードマップ候補)

| 段階 | 施策 | 価値 |
|------|------|------|
| Near | タイマーのリロード耐性 | 信頼性向上 |
| Near | PWA化 / オフライン | モバイル常用化 |
| Mid  | マルチユーザー + 認証 | チーム利用 / 課金導線 |
| Mid  | カレンダー連携 (Google/iCal) | 既存ワークフロー接続 |
| Mid  | AIによるタスク分解 / 見積補助 | 差別化 |
| Far  | Slack / 通知連携、レポート出力 | 顧客LTV向上 |

## 競合ポジショニング

|          | TODO管理 | 時間計測 | 排他タイマー | 一画面集約 |
|----------|:--------:|:--------:|:------------:|:----------:|
| Todoist  | ◎        | △        | ✕            | ✕          |
| Toggl    | △        | ◎        | ✕            | ✕          |
| Notion   | ○        | ✕        | ✕            | ○          |
| **OneLogue** | ○    | ○        | **◎**        | **◎**      |

差別化軸は「排他タイマー × TODO統合 × 単一画面」。Toggl + Todoistを往復する層が明確なターゲット。

## 技術的な見どころ

- **Server/Client時刻同期**: layout で server 時刻を Provider にシード、クライアントは `setInterval` で更新。hydration整合性を保ちつつリアルタイム表示。
- **Tokyo固定のカレンダー算術**: UTC基準 + 9時間オフセットでローカル依存を排除した日付計算ユーティリティ群 (`lib/datetime.ts`)。
- **排他タイマーの一貫性**: `/tasks/{id}/start` が既存計測を自動停止する原子的処理。
